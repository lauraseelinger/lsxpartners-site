The Blueprint
The foundation that makes RxSight’s own pages the source AI cites.
The sitemap fixed where things live. This fixes how each page is built so AI can read it and pull from it.
How this pairs with the sitemap
The sitemap is the IA layer — the nav tree, what’s a page vs. a spoke. This Blueprint is the layer underneath: the foundation (site-wide), the universal page rules, and a template per page type.
How to read it
  • Universal page rules — the rules every page follows, each with a diagram.
  • Foundation — the site-wide work the templates sit on.
  • Page templates — the sidebar lists every page type. Pick one → its spec + the patient questions it must answer + a layout mockup of where the work goes.
  • Blog — the Patient Resources content: the article template, the live/planned/backlog library, and the launch bucket.
What’s “new” vs. “make AI-ready”
Some pages exist today (Homepage, Patients landing, How the LAL Works, Post-op guide, FAQ) — the work is making them AI-readable: front-loaded answers, extractable text, schema. The genuinely net-new pieces are the lens-decision shelf, the comparison content, the problem-aware symptom cluster, and the maker/entity page. Each template is tagged ADD, AI OPTIMIZED, or RECOMMEND.
Universal page rules
The rules every page follows — before any template-specific work.
Get these right on every page and you’ve fixed most of the retrieval gap. The templates assume all of this is in place.
1 · Front-load the answer. The first ~100–200 words carry the citation weight — write a self-contained 50–100-word answer to the page’s primary patient question at the very top. i
◤ ANSWER — the patient’s question, answered in the first 100 words
↑ answer up top · brand depth below
2 · Make it extractable. Real text in HTML — never lock the answer, the how-it-works steps, or claims in PDFs/images. i
✗ PDF / image
patient-guide.pdf
✓ real HTML text
3 · Headings = the questions patients ask. One H1; H2s phrased as the real patient question (“Why is my vision blurry after cataract surgery?”), H3s for subtopics. i
H1 · Life After Cataract Surgery
H2 · Why is my vision blurry after surgery?
H3 · When does it improve?
H2 · Can it be corrected?
4 · Internal linking, hub-and-spoke. Descriptive anchors (never “click here”). The hub links down to its spokes; every spoke links back up; symptom pages route to the lens that can be adjusted. i

Foundation
The site-wide work the templates sit on.
Get these wrong and the per-page work leaks. Stand them up early so pages launch citable, not retrofitted.
Schema / structured-data strategy — the per-template schema set applied site-wide; the LAL described as a MedicalDevice with RxSight as manufacturer everywhere it appears.

Entity / brand-signal hygiene — the About / Organization page makes AI attribute the Light Adjustable Lens to RxSight. Patients know the product, not the maker (named in 28 of 326 captured threads) — this closes that gap so engines stop confusing or omitting it. i
✓ Already in place: /about/ exists with Organization schema — so this is a refine, not a from-scratch build.
→ To do:
• AI names the LAL but credits RxSight as the maker in only 39 of 80 answers. → State plainly on /about/ + product pages that “RxSight manufactures the Light Adjustable Lens,” with Organization + manufacturer schema and sameAs links (LinkedIn, the RXST investor page, and any authoritative directories where RxSight is listed).
• The former name “Calhoun Vision” still surfaces. → Make the current RxSight entity unambiguous so engines resolve to it, not the old name.
• The LAL gets miscategorized as “a monofocal you pay to adjust.” → Define it correctly as an adjustable IOL (DefinedTerm) on the entity + how-it-works pages.
• ActivShield is rarely tied to RxSight. → Name it on the entity / product pages.
Re-verify each against live AI answers at build.
Technical extractability — keep every answer page as crawlable, semantic HTML (no text-in-images); clean sitemap.xml + robots + Search Console; add an llms.txt pointing AI crawlers at the priority answer pages (emerging signal, low cost).
✓ Already in place: sitemap.xml is live and the site is already HTML-based — the only PDF is the regulatory SSED (fine to leave). No patient or clinical guides are locked in PDFs, so there’s nothing to convert.
→ To add: an llms.txt (currently returns 404). That’s it on extractability — the content is already HTML.
Migration discipline / canonical — preserve the ~1,248 organic keyword rankings (Semrush US, Jun 2026) through the rebuild: 301-redirect every changed URL and keep the on-page content that earns each ranking. One canonical URL per topic — where two pages target the same query (e.g. a blog post and a hub section both on “LAL cost”), set rel=canonical on the secondary → primary or merge them, so engines don’t split the signal or cite a thin duplicate; give the /de/ /es/ /ja/ etc. versions hreflang so they read as translations, not duplicates. i
Find-a-Doctor structured data — Physician / LocalBusiness schema so engines answer “LAL near me,” and the locator stays the conversion surface; protect the existing local-ranking strength.
